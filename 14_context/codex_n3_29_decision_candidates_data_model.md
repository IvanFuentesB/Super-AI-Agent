# Codex N+3.29 Decision Candidates Data Model

Status: codex_planning_only / decision_candidates_model / not_runtime_wired

Date: 2026-04-29
Branch: feat/ghoti-visible-operator-stack

## Purpose

Define the `decisions_recommended.jsonl` draft artifact produced by the future weekly review generator. These are candidate recommendations only. They are not appended to `manual_decision_queue.jsonl` and they do not execute any action.

## Artifact Path

Future artifact:

```text
05_logs/money_reviews/<run_id>/decisions_recommended.jsonl
```

## Required Fields

Each line should be one JSON object with:

- `decision_id`
- `created_at`
- `source_run_id`
- `experiment_id`
- `decision_type`
- `confidence`
- `reason`
- `suggested_next_action`
- `approval_required`
- `status`
- `risk_level`
- `files`
- `notes`

## Decision Types

Allowed values:

- `DOUBLE_DOWN`
- `ITERATE`
- `PAUSE`
- `KILL`
- `BUILD_NEXT`
- `CREATE_CONTENT_BATCH`
- `CREATE_LEAD_MAGNET`
- `REVIEW_LAUNCH_CHECKLIST`
- `COLLECT_MORE_DATA`

## Status

All generated records must use:

```json
"status": "candidate"
```

The weekly review generator must not mark candidates as accepted, in progress, completed, killed, or superseded. Those statuses belong to the future manually reviewed queue.

## Confidence

Allowed values:

- `low`
- `medium`
- `high`

Use `low` when:

- no metrics exist
- only sample records exist
- scoring is missing
- claims/proof is unclear

Use `medium` when:

- product/customer/offer are clear
- distribution path exists
- some manual metrics exist

Use `high` only when:

- manual metrics support the recommendation
- risk is low or clearly gated
- next manual action is specific

## Validation Rules

The future implementation should:

- parse candidate JSONL line by line
- reject non-object lines
- require all fields above
- require enum values for `decision_type`, `confidence`, `status`, and `risk_level`
- force `status` to `candidate`
- force `approval_required` to true for public/money-facing next actions
- reject candidates that imply automatic live actions
- cap candidates to a practical number, such as 20
- write invalid lines to `parse_warnings.json`

## Relationship To Manual Decision Queue

`decisions_recommended.jsonl`:

- generated by Gemma
- draft-only
- may contain imperfect ideas
- stored under `05_logs/money_reviews/<run_id>/`
- never executed
- never auto-appended

`manual_decision_queue.jsonl`:

- future operator-reviewed queue
- append-only
- records accepted manual decisions
- created by explicit operator command or future approved helper

The transition from candidate to queue should always require operator review.

## Safety Rejections

The future parser should flag or reject candidates that say:

- post automatically
- send email automatically
- DM people
- scrape platform metrics
- upload listing
- activate payment
- log into account
- fake testimonial
- fake scarcity
- fake income proof
- buy followers/views/comments
- bypass platform limits

## Verdict

Decision candidates should make the weekly review actionable without removing the operator from control.
